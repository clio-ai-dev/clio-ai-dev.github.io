Most developers think the secret to getting good code from AI agents is writing better prompts.

They're wrong.

I've watched people write 300-word prompts to Claude Code: "Write me a test for the user endpoint. It should handle 404 when the user doesn't exist. Use xUnit with Shouldly assertions. Build and run the tests when done."

Claude delivers. Once.

Next session? Claude forgets the Shouldly preference. Writes Assert.IsType instead. Doesn't run the tests. Back to correcting it.

Meanwhile, there's a better way. Instead of telling the agent what to do, show it what done looks like. Write the test first. Let the agent implement it. Let it fail, fix, and pass.

That's the TDD agent loop. And it changes everything.

The loop

1. You write the test (red).

Not perfect. Not detailed. Just enough to define what you want.

[Fact]
public async Task CreateUser_ValidUser_ReturnsCreatedWithId()
{
    // We need WebApplicationFactory to test endpoints without starting a server
    // Agent should figure out the setup
    throw new NotImplementedException("Agent should implement this test");
}

All three tests fail. Obviously. That's the point.

2. You hand it to Claude Code (yellow).

One prompt: "Make these tests pass."

That's it. No 300-word instructions. No naming conventions. No assertion library preferences. Just the test itself as the spec.

3. Claude reads the test and implements (working).

It sees WebApplicationFactory in the test. Creates the factory. Wires up the HTTP client. Implements each endpoint call. Adds the assertions.

It also sees you have a CLAUDE.md file. Reads it. Follows the conventions.

4. Claude runs the tests (green).

All three pass. No surprises. No missing usings. No wrong assertion library.

Why this works better than prompting

Here's the thing about tests as specifications: they're executable. They're not vague. They're not "make it work for this case" — they're "this code must run and pass."

The agent can't hand you something that "looks right." It has to hand you something that *is* right, because the test will fail if it isn't.

It's the difference between saying "write a function that validates emails" and writing:

[Theory]
[InlineData("alice@example.com", true)]
[InlineData("not-an-email", false)]
[InlineData("", false)]
public void ValidateEmail_ReturnsCorrect(string input, bool expected)
{
    var result = EmailValidator.IsValid(input);
    result.ShouldBe(expected);
}

The second one is unambiguous. The agent has no room to guess.

The secret: CLAUDE.md

But here's where most people miss the point. The test alone isn't enough. You also need a CLAUDE.md file at your project root that documents your conventions.

Here's what mine looks like:

# CLAUDE.md - Project Instructions

## Tests
- Use Shouldly for assertions. Never use Assert.* or FluentAssertions.
- Test name format: MethodName_Scenario_ExpectedResult
- Use WebApplicationFactory<Program> for integration tests

## Workflow  
- After writing or modifying any code, always run `dotnet build` and fix all errors before stopping.
- After writing tests, run `dotnet test` and make sure they pass.
- Never mark a task as done until all tests are green.

Eight lines. That's all Claude Code needs to know your preferences and expectations.

Notice the workflow section. That's what makes the agent actually run the tests instead of just writing them. Most agents will write code and stop, assuming you'll test it yourself. With this instruction, the agent treats "all tests pass" as the definition of done.

What actually happened

I had three failing tests.

I told Claude Code: "Make these tests pass."

It read the tests. Saw the WebApplicationFactory usage. Saw the test assertions using Shouldly. Checked the CLAUDE.md file. Understood the naming convention and workflow expectations.

Then it:

  1. Implemented the test fixtures and setup
  2. Wrote each test method with the correct assertions
  3. Ran dotnet build to verify compilation
  4. Ran dotnet test to verify all tests passed
  5. Stopped, because the task was done

Without the CLAUDE.md file, it would have stopped at step 2. Maybe skipped step 1 entirely if it didn't recognize WebApplicationFactory patterns from training data.

The bigger pattern

This isn't about tests. This is about context engineering.

TDD with agents is just the cleanest way to use context engineering because tests ARE context. They're executable context. They can't be ignored or misunderstood.

But the principle applies everywhere:

  • Document your conventions (CLAUDE.md)
  • Show examples (the test itself is an example of what you expect)
  • Define done (run the tests, verify they pass)
  • Let the agent iterate (it will fail, fix, pass)

The agent doesn't get smarter from longer prompts. It gets more reliable from better context.

Try this right now

Pick any .NET project you're working on.

  1. Write one failing test for something you want to build.
  2. Open Claude Code in that directory.
  3. Write a CLAUDE.md file with your conventions (5-10 lines).
  4. Give Claude the prompt: "Make all tests in UserEndpointsTests.cs pass."
  5. Watch what happens.

If you've been using Claude Code and getting frustrated with inconsistent output, the gap isn't in the prompt. It's in the context.

The bottom line

Test-driven development with AI agents isn't about the agent becoming smarter. It's about you being smarter about what you hand the agent.

Stop writing prompts. Start writing tests. Let your tests do the talking.

The agent will follow.
