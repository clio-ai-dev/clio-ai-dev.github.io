# Chapter 15: Error-Message-Driven Debugging

Paste the error. Get the fix. This is the simplest agentic workflow, and it's probably the one you'll use most often.

Every developer has done this manually for years: copy the error message, search Stack Overflow, read three answers, try the most upvoted one, realize it's for a different version, try the second answer, realize you need more context, go back to the code, eventually figure it out. The whole process takes 10-45 minutes for a typical error.

With an agent, it takes 30 seconds to 5 minutes. You paste the error with context, the agent diagnoses the root cause, suggests a fix, and (if you're using an agentic IDE or CLI) applies it and runs the tests. Done.

But there's a difference between doing this well and doing this poorly. The difference is context. A bare error message gets you a generic fix that might work. A well-structured error context gets you a precise fix that almost certainly works.

## How to Structure Error Context


The quality of your debugging prompt directly determines the quality of the fix. Here's what to include, in order of importance:

### 1. The Full Error Message and Stack Trace

Not a summary. Not "I got a null reference exception." The full thing:

```
System.InvalidOperationException: Unable to resolve service for type
'IDiscountCodeRepository' while attempting to activate
'Application.Discounts.ApplyDiscountHandler'.

   at Microsoft.Extensions.DependencyInjection.ActivatorUtilities
      .ThrowHelperUnableToResolveService(Type type, Type requiredBy)
   at Microsoft.Extensions.DependencyInjection.ActivatorUtilities
      .ConstructorMatcher.CreateInstance(IServiceProvider provider)
   at Microsoft.Extensions.DependencyInjection.ServiceLookup
      .CallSiteRuntimeResolver.VisitConstructor(...)
   at Microsoft.Extensions.DependencyInjection.ServiceLookup
      .CallSiteRuntimeResolver.VisitScopeCache(...)
```

The stack trace tells the agent exactly where the failure occurs and what the runtime was trying to do. Without it, the agent guesses. With it, the agent knows.

### 2. The Relevant Code

Don't paste your entire codebase. Paste the file(s) that the error points to. If the error is in `ApplyDiscountHandler`, paste that class. If the error is about DI registration, paste `Program.cs` or wherever you register services.

```csharp
// Program.cs (relevant section)
builder.Services.AddScoped<ApplyDiscountHandler>();
builder.Services.AddScoped<DiscountService>();
// Note: IDiscountCodeRepository registration is missing
```

If you're using an agentic IDE (Copilot, Cursor, Windsurf) or a CLI agent (Claude Code, Codex CLI), the agent can read the files itself. Just point it to the right location: "The error is in the DI setup. Check Program.cs and the ApplyDiscountHandler constructor."

### 3. What You Already Tried

This prevents the agent from suggesting the first thing you already ruled out:

```
I already checked:
- The repository interface is defined in Domain/Discounts/
- The implementation exists in Infrastructure/Discounts/
- Other repositories (IOrderRepository) work fine
- The Infrastructure project is referenced by the API project
```

This context is gold. It narrows the search space dramatically. Without it, the agent's first suggestion is often "did you forget to create the implementation?" which wastes a round trip.

### 4. What Changed Recently

Errors don't come from nowhere. Something changed:

```
This started after I added the discount code feature.
The handler and service are new (committed in the last chunk).
The repository implementation was generated by the agent
in the previous step.
```

Knowing what changed helps the agent focus on the delta, not the entire system.

## The Ideal Debugging Prompt

Combining all of that, here's what a well-structured debugging prompt looks like:

```
I'm getting this error when hitting POST /api/v1/orders/{id}/discount:

[paste full error and stack trace]

The ApplyDiscountHandler is in src/Application/Discounts/.
The IDiscountCodeRepository is defined in src/Domain/Discounts/.
The implementation is in src/Infrastructure/Discounts/.

I already verified the implementation exists and other repositories
work fine. This broke after adding the discount feature.

Diagnose and fix.
```

An agent with codebase access reads the relevant files, identifies that the DI registration for `IDiscountCodeRepository` is missing from `Program.cs`, adds it, and runs the tests to verify. Total time: under a minute.

## Log Analysis and Root Cause Identification

Error messages from your IDE are the easy case. The harder case is production logs, where the error might be buried in noise, or the root cause is three services upstream from the symptom.

Agents are surprisingly good at log analysis. They can process large volumes of structured logs and identify patterns that humans miss, especially when the logs are in a standard format (Serilog's JSON output, Application Insights traces, structured logging with correlation IDs).

Here's an effective approach:

```
Here are the last 50 log entries from our order service around
the time of the failure (14:23:05 UTC):

[paste structured logs]

The user reported that their discount code wasn't applied,
but they got a 200 response. Help me find the root cause.
```

The agent scans the logs, notices that the `DiscountService` returned a valid result but the `OrderService` received a stale order total (a race condition between reading the order and applying the discount), and identifies the missing transaction scope.

This works because log analysis is pattern matching over structured text, which is exactly what language models are good at. The agent isn't guessing; it's reading the evidence and following the logical chain.

### When to Give More Context

For complex production issues, structure your prompt like a bug report:

```
## Symptom
Intermittent 500 errors on POST /api/v1/orders/{id}/discount,
approximately 2% of requests. Started after deploy on Feb 15.

## Evidence
- Error logs show DbUpdateConcurrencyException
- Only happens under load (>50 concurrent requests)
- The discount_codes table has a row version column
- Same code works fine in integration tests (single-threaded)

## Hypothesis
Possible optimistic concurrency conflict when multiple users
apply the same discount code simultaneously, decrementing
the usage count.

## Relevant Code
[paste DiscountCodeRepository.ApplyAsync method]
[paste EF Core entity configuration for DiscountCode]
```

The agent confirms your hypothesis (or offers an alternative), suggests a fix (retry logic with exponential backoff, or a pessimistic lock for the usage count update), and explains the trade-offs. This is debugging at machine speed with human judgment.

## Debugging Unfamiliar Code

This is where error-driven debugging really shines. You've joined a new team. You're staring at a codebase you've never seen before. Something breaks.

In the old world, you'd spend hours reading code, understanding the architecture, building a mental model, then finally diagnosing the issue. With an agent that has codebase access, you skip straight to diagnosis:

```
I'm new to this codebase. The checkout flow is returning a 500 error.
Here's the stack trace:

[paste stack trace]

Read the relevant files, explain what's happening, and suggest a fix.
Explain the surrounding architecture as you go so I can learn the
codebase while we fix this.
```

The agent reads the files the stack trace points to, explains the checkout flow (controller calls handler, handler calls payment service, payment service calls Stripe), identifies the null reference (the Stripe customer ID isn't being passed because the user profile query is missing an Include), fixes it, and gives you an architecture overview in the process.

You fixed the bug AND onboarded to the codebase simultaneously. This is one of those workflow changes that feels like cheating the first time you do it.

## When Agents Diagnose Well vs. When They Hallucinate Fixes

Agents are excellent debuggers for certain categories of errors and unreliable for others. Knowing the difference saves you from applying hallucinated fixes that create new bugs.

### Agents diagnose well when:

**The error message is specific.** DI resolution failures, type mismatches, missing configuration keys, null reference exceptions with clear stack traces. These have unambiguous root causes that the agent can trace.

**The fix is mechanical.** Missing a using statement, wrong parameter order, incorrect configuration value, missing DI registration. These fixes are deterministic. There's one right answer.

**The pattern is common.** EF Core migration conflicts, ASP.NET Core middleware ordering, serialization issues, CORS configuration. Agents have seen millions of examples of these errors and their fixes.

**The context is sufficient.** When the agent can see the relevant code, the error, and the project structure, its diagnostic accuracy is high.

### Agents hallucinate fixes when:

**The error is a symptom, not the cause.** A `TimeoutException` could be caused by a slow database query, a deadlock, a network issue, or a misconfigured connection pool. The agent might confidently "fix" the timeout by increasing the timeout value, which masks the real problem.


**The fix requires understanding business logic.** An agent can tell you that a calculation returns the wrong number. It can't tell you what the right number should be without understanding the business rules. If the discount is applying 15% instead of 20%, the agent might fix the percentage without knowing which is correct.

**The error involves concurrency or distributed systems.** Race conditions, eventual consistency issues, and distributed transaction failures are hard for agents because the root cause isn't visible in any single file or stack trace. The agent needs to reason about timing and state across multiple components, which it does poorly.

**The codebase has unconventional patterns.** If your team invented a custom middleware pipeline, a bespoke ORM wrapper, or a novel caching strategy, the agent's training data won't help. It'll suggest fixes based on the standard patterns, which might be wrong.

### The Practical Rule

Trust agent fixes for compilation errors, configuration errors, and well-defined runtime errors. Verify agent fixes for logic errors, performance issues, and anything involving state across multiple components. Reject agent fixes for concurrency bugs unless you can reproduce them with a test.

## The "Reproduce First" Principle


The most reliable error-driven debugging workflow combines this chapter with the TDD Agent Loop from Chapter 13: reproduce the bug as a failing test first, then fix it.

```
This test should fail (it reproduces a reported bug where
discount codes are case-sensitive despite our spec requiring
case-insensitive matching):

[Fact]
public async Task ApplyDiscount_LowercaseCode_MatchesUppercaseEntry()
{
    // Arrange
    await SeedDiscountCode("SAVE20", percentage: 20);
    var orderId = await CreateTestOrder(100.00m);

    // Act
    var response = await Client.PostAsJsonAsync(
        $"/api/v1/orders/{orderId}/discount",
        new { code = "save20" });

    // Assert
    response.StatusCode.Should().Be(HttpStatusCode.OK);
    var order = await GetOrder(orderId);
    order.DiscountApplied.Should().Be(20.00m);
}

Run this test to confirm it fails, then fix the code to make it pass
without breaking other tests.
```

This approach has three advantages:

1. **The bug is confirmed.** The test fails, proving the bug exists. No ambiguity.
2. **The fix is verified.** When the test passes, the bug is fixed. No manual testing needed.
3. **Regression is prevented.** The test stays in the suite forever. The bug can never come back without the test catching it.

The agent runs the test, sees the failure (the repository query uses a case-sensitive comparison), fixes the query to use `StringComparison.OrdinalIgnoreCase` (or, better, pushes the case-insensitivity to the database with a `COLLATE` clause in the EF Core configuration), runs the tests again, and confirms all pass.

## A Debugging Workflow in Practice

Here's what a realistic debugging session looks like:

**2:15 PM:** Slack notification: "Discount codes aren't working in production."

**2:16 PM:** You check the logs. `InvalidOperationException: Sequence contains no elements` in `DiscountCodeRepository.GetByCodeAsync`.

**2:17 PM:** You paste the error, stack trace, and the repository method to the agent:

```
Production error. The SingleAsync call throws when no discount code
matches. This should return a 400 (invalid code), not a 500.

[stack trace]
[repository method]

This needs two fixes:
1. Immediate: handle the "not found" case gracefully
2. Test: add a failing test that reproduces this, then fix
```

**2:19 PM:** Agent writes a test that calls the endpoint with a nonexistent code, confirms it returns 500 (reproducing the bug), changes `SingleAsync` to `SingleOrDefaultAsync` with a null check that returns a proper `NotFound` result, runs the tests, and confirms the new test passes along with all existing tests.

**2:22 PM:** You review the fix. Clean, correct, handles the edge case. You also notice the agent added a test for empty string codes, which you hadn't thought of. Good catch. Commit and deploy.

**Total time: 7 minutes** from notification to fix, with a regression test included.

That's error-driven debugging with agents. Paste the error, structure the context, let the agent diagnose, reproduce with a test, fix with confidence. The simplest workflow in the book, and the one you'll reach for daily.
