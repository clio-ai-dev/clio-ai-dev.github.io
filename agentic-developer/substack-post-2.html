Most developers think the secret to getting good code from AI is writing better prompts.

It's not.

I've watched people craft 200-word prompts full of instructions. Every. Single. Time. They describe the architecture, the naming conventions, the test framework, the folder structure. And they get decent output. Once.

Next session? The agent forgets everything. Back to square one. Another 200-word prompt.

Meanwhile, I've seen developers write "add a test for the GetUserOrders endpoint" and get production-ready code back. Correct conventions. Right assertion library. Tests that actually compile and pass.

The difference? Context.

Not the prompt. The context file sitting in the project root that the agent reads automatically every time it starts.

Let me show you exactly what I mean.

Same prompt. Wildly different output.

I ran an experiment with Claude Code on a .NET project. Same prompt both times: "Write a test for the GetUserOrders endpoint. It should return 404 when the user doesn't exist."

Same model. Same codebase. The only difference: whether a CLAUDE.md file existed in the project root.

Without a context file:

[Fact]
public async Task GetUserOrders_ReturnsNotFound_WhenUserDoesNotExist()
{
    // ...setup...

    var result = await OrdersEndpoints.GetUserOrders(userId: 99, db);

    Assert.IsType<NotFound>(result);
}

Three problems here. First, it used Assert.IsType, plain xUnit. My project uses Shouldly. Second, the test name format is wrong. I use MethodName_Scenario_ExpectedResult, not MethodName_ReturnsX_WhenY. Third, and this is the big one: the agent didn't build or run the test. It just wrote it and stopped.

So I tried building it myself:

It doesn't even compile. The agent wrote code, declared it done, and moved on. Missing usings, missing package references. The kind of thing you'd catch in 10 seconds if you just ran dotnet build.

public async Task GetUserOrders_UserDoesNotExist_ReturnsNotFound()

    result.ShouldBeOfType<NotFound>();
}

Shouldly assertion. Correct test name format. And the agent built the project and ran the test before finishing:

The agent didn't get smarter. It got better informed.

Why this happens

AI coding agents are stateless. Every session starts from zero. The agent reads your codebase, makes its best guess based on training data, and produces output.

The problem? Training data is dominated by the most popular patterns. For .NET, that means Assert.IsType because that's what every xUnit tutorial uses. It means ReturnsNotFound_WhenUserDoesNotExist because that naming format is everywhere on Stack Overflow. It means the agent won't build or run anything unless you tell it to, because most code examples online are snippets, not verified working code.

Without explicit guidance, the agent defaults to the internet's conventions. Not yours.

And here's what makes it worse: this compounds across sessions. The first session might be fine because you're there, correcting things in real time. But session 5? Session 20? Every new session starts cold. Every time, the agent makes the same wrong assumptions. Every time, you catch the same mistakes in review.

Context engineering fixes this permanently.

What context engineering actually looks like

Context engineering is giving your AI agent the right information before it starts working, so you don't have to fix it after.

It's not a new framework. It's not a plugin. It's a markdown file in your project root that the agent reads automatically.

Here's the stack, from highest leverage to lowest.

Layer 1: Your tool's context file

Start here. A single file at your project root. Documents your conventions, architecture decisions, and what NOT to do. Loads automatically every session.

Each tool has its own file:

  • Claude Code: CLAUDE.md
  • GitHub Copilot: .github/copilot-instructions.md
  • Cursor: .cursor/rules/

Different names. Same idea. You write it once, the agent reads it every time.

Here's what mine looks like:

## Tests
- Shouldly for assertions. Never Assert.* or FluentAssertions.
- Test name: MethodName_Scenario_ExpectedResult

## Workflow
- After writing or modifying any code, always build and fix
  all errors before stopping.
- After writing tests, run them and make sure they pass.

Eight lines. That's it. The agent went from wrong naming, wrong assertion library, and no build verification to all three correct.

Notice the workflow section. That's what made the agent actually run dotnet build instead of just writing code and walking away. Without it, the agent assumes "write code" is the end of the task. With it, the agent treats "code that compiles and passes" as the end of the task.

That single instruction, "always build and fix all errors before stopping", is the difference between code you review and code you debug.

Layer 2: Path-scoped rules

As your context file grows, you'll notice some rules only matter for certain files. Test conventions are irrelevant when the agent is editing an API endpoint. API patterns are noise when the agent is writing tests.

Path-scoped rules solve this. They load only when the agent touches matching files.

In Claude Code, you break instructions into separate files under .claude/rules/:

your-project/
├── .claude/
│   ├── CLAUDE.md              # Project-wide instructions
│   └── rules/
│       ├── api-endpoints.md   # API conventions
│       ├── testing.md         # Test conventions
│       └── security.md        # Security requirements

Each rule file can target specific paths using frontmatter:

---
paths: src/**/Endpoints/*.cs
---
- All endpoints require .RequireAuthorization() unless explicitly public.
- Return typed Results (Results.Ok, Results.NotFound). Never return raw objects.
- Map to response records inline. No DTOs in separate files.

This rule loads only when the agent edits endpoint files. When the agent is writing tests, these rules don't exist. Context stays focused, and the agent doesn't get confused by irrelevant instructions.

GitHub Copilot and Cursor support the same idea in their own locations (.github/instructions/ and .cursor/rules/).

Layer 3: The "don'ts"

This is the one most developers miss. And honestly, it might be the most important layer.

What the agent should NOT do is more valuable than what it should do.

Why? Because without explicit constraints, agents default to popular patterns from training data. And popular doesn't mean good. It means common. The patterns that show up in the most tutorials, the most Stack Overflow answers, the most blog posts.

For a .NET project, that means the agent will:

  • Scaffold a Repositories/ folder with interfaces and implementations
  • Add AutoMapper for mapping between models
  • Wire up MediatR for request handling
  • Generate controllers instead of minimal API endpoints
  • Layer everything into Clean Architecture folders

None of that is how I work. And if I don't say so explicitly, the agent will keep doing it. Every single session.

So I tell it:

- No repository pattern. Inject DbContext directly.
- No AutoMapper or MediatR. Map manually.
- No clean architecture. Vertical slices only.
- No controllers. Minimal APIs with static endpoint methods.
- No CQRS. It's a web API, not an event-sourced system.

Every line here exists because the agent kept doing it until I told it not to. Now it doesn't.

Think of it this way: your positive instructions tell the agent what to build. Your negative instructions tell it what NOT to build. Without both, the agent fills in the gaps with Stack Overflow's greatest hits.

4 mistakes that kill your context files

I've seen these patterns kill context files over and over.

The Novel. A 2,000-line file documenting every class and method. Too much context actually degrades model output. The agent gets overwhelmed and starts ignoring instructions. Keep it under 200 lines. The agent can read your source code when it needs specifics. Your context file is for conventions and decisions, not documentation.

The Museum. Written once, never updated. Your project evolves. New patterns emerge. Old patterns get deprecated. If your context file still references patterns you abandoned six months ago, the agent is following outdated instructions. Review it every couple of weeks.

The Wishlist. Full of conventions your codebase doesn't actually follow. If you write "always use records for DTOs" but half your codebase uses classes, the agent sees the contradiction and picks whichever it prefers. Document what IS, not what you wish.

The Secret Keeper. Has positive instructions but no negative constraints. You decided against repository pattern months ago, but you never wrote it down. So every new session, the agent suggests repositories again. Then you correct it. Then next session, it suggests them again. Write down what you decided against and why. The agent will stop suggesting it.

Try this right now

You don't need to spend a day on this. 30 minutes is enough to see the difference.

  1. Create your tool's context file at your project root (CLAUDE.md, .github/copilot-instructions.md, or .cursor/rules/).
  2. Write 10-15 lines covering your test framework, naming convention, workflow expectations (build? run tests? fix errors?), and 3-5 things the agent should NEVER do.
  3. Give the agent the same prompt you gave it yesterday. Compare the output.

If you want to try this yourself, pick any project you're working on. Delete your context file (or start without one), ask the agent to write a test. Then add the file and ask again. The difference will be obvious.

The bottom line

Your prompt is temporary. Your context file is permanent.

Every convention you document is a convention the agent follows automatically instead of you catching it in review. Every "don't" you write is a bad suggestion you'll never see again. Every workflow instruction is a step the agent takes without you having to ask.

Context engineering doesn't just help. It compounds. Every line you add makes every future session more productive. And unlike your prompts, it survives across sessions, across team members, and across tools.

Stop writing better prompts. Write better context.
